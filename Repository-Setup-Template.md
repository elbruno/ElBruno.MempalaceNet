**Key Notes:**
- `docs/` contains all documentation, images, and assets
- `src/` contains implementation only
- `tests/` is separate from src
- `.github/workflows/` contains CI/CD automation
- `.squad/` is optional (for team-based projects)
- Symlinks allow `README.md`, `LICENSE`, `CHANGELOG.md` at both root AND docs/ for convenience

## Code Organization

### .csproj File Structure

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <TargetFramework>net8.0</TargetFramework>
    <LangVersion>12</LangVersion>
    <Nullable>enable</Nullable>
    
    <!-- Package Metadata -->
    <PackageId>ElBruno.{ProjectName}</PackageId>
    <Version>0.1.0</Version>
    <Title>ElBruno.{ProjectName}</Title>
    <Authors>Bruno Garcia;ElBruno</Authors>
    <Description>Short description of the project purpose</Description>
    <PackageProjectUrl>https://github.com/elbruno/ElBruno.{ProjectName}</PackageProjectUrl>
    <PackageIcon>nuget-icon-128x128.png</PackageIcon>
    <RepositoryUrl>https://github.com/elbruno/ElBruno.{ProjectName}</RepositoryUrl>
    <RepositoryType>git</RepositoryType>
    <PackageTags>tag1;tag2;tag3</PackageTags>
    <PackageLicenseExpression>MIT</PackageLicenseExpression>
    
    <!-- README -->
    <PackageReadmeFile>README.md</PackageReadmeFile>
  </PropertyGroup>

  <!-- Icon Asset (CRITICAL: must exist in git) -->
  <ItemGroup>
    <None Include="../../docs/images/nuget-icon-128x128.png" Pack="true" PackagePath="" />
    <None Include="../../README.md" Pack="true" PackagePath="" />
  </ItemGroup>

  <!-- Dependencies (minimal by default) -->
  <ItemGroup>
    <!-- Add specific dependencies only when needed -->
  </ItemGroup>

</Project>
```

### C# Coding Standards

- **Null-safe C#:** Always set `<Nullable>enable</Nullable>` in .csproj
- **Modern features:** Use C# 12+ features (records, init-only properties, required keyword, etc.)
- **Naming:** PascalCase for public APIs, _camelCase for private fields
- **Comments:** Only comment non-obvious logic. Self-documenting code is preferred.
- **No external dependencies unless essential:** Keep projects lightweight
- **Unit tests required:** Every public method should have test coverage

### File Naming Conventions

- Public classes: `{ClassName}.cs` (one per file if possible)
- Internal/private classes: `{ClassName}.Internal.cs` or grouped in feature folders
- Tests: `{TestedClassName}Tests.cs`
- Interfaces: `I{InterfaceName}.cs`

## NuGet Publishing Process

### Critical Lessons Learned (from Real-World Releases)

**DO THIS FIRST** — Before attempting your first NuGet publish:

1. ✅ **Workflow filename MUST be exactly `publish.yml`** (not `publish-nuget.yml` or other variants)
   - NuGet trusted publishing policy enforces this filename match
   - Wrong filename → `Workflow mismatch for policy` error
   - **Verified in ElBruno.BM25 v0.5.1 release**
   - If you rename the file, **DELETE and RECREATE the git tag** (old tag points to old commit with wrong filename)

2. ✅ **Use OIDC-based trusted publishing (NOT API keys)**
   - Requires NuGet trusted publishing policy entry at NuGet.org dashboard
   - More secure than API keys
   - Requires matching service principal username on both GitHub and NuGet
   - **Tested successfully in v0.5.1 release (no API keys needed)**

3. ✅ **Commit all package assets BEFORE creating the release tag**
   - Icons, images, promotional assets must exist in git
   - If they don't, `dotnet pack` fails during CI/CD
   - If you add assets after creating a tag, **DELETE and RECREATE the tag**
   - **v0.5.1: All CHANGELOG.md and docs assets committed before tag creation**

4. ✅ **Use the EXACT OIDC pattern** (reference: `github.com/elbruno/ElBruno.Text2Image`)
   - `id: login` on NuGet/login step to capture output
   - `${{ secrets.NUGET_USER }}` (not hardcoded username)
   - `--api-key ${{ steps.login.outputs.NUGET_API_KEY }}` on nuget push

5. ✅ **Fix .csproj metadata BEFORE packing**
   - Replace deprecated `<License>` with `<PackageLicenseExpression>` (e.g., `MIT` or `Apache-2.0`)
   - Update package metadata in .csproj: Version, PackageId, Authors, Description, RepositoryUrl, PackageProjectUrl
   - **v0.5.1 actual fix**: Changed `<License>` to `<PackageLicenseExpression>MIT</PackageLicenseExpression>`
   - NuGet pack fails if metadata violates NuGet schema

### Successful Release Process (v0.5.1 Example)

**Step 1: Prepare .csproj metadata**
```xml
<PropertyGroup>
  <Version>0.5.1</Version>
  <PackageId>ElBruno.BM25</PackageId>
  <Authors>ElBruno</Authors>
  <Description>BM25 Full-Text Search Library for .NET</Description>
  <PackageLicenseExpression>MIT</PackageLicenseExpression>
  <RepositoryUrl>https://github.com/ElBruno/ElBruno.BM25</RepositoryUrl>
  <PackageProjectUrl>https://github.com/ElBruno/ElBruno.BM25</PackageProjectUrl>
</PropertyGroup>
```

**Step 2: Commit all assets**
```bash
# Include updated CHANGELOG.md, README, docs, icons, etc.
git add docs/ CHANGELOG.md src/ElBruno.BM25/*.csproj
git commit -m "docs: prepare v0.5.1 release"
```

**Step 3: Create version tag (v{VERSION} format)**
```bash
git tag v0.5.1
git push origin v0.5.1
```

**Step 4: Monitor publish.yml workflow**
- GitHub Actions triggers automatically on tag push
- Workflow uses OIDC token exchange (via NuGet/login@v1 with id: login)
- Package published to NuGet.org when workflow completes

**Result:** v0.5.1 published to NuGet in 2-3 minutes, zero manual API key entry.

### Workflow File Template

**File:** `.github/workflows/publish.yml`

```yaml
name: Publish to NuGet

on:
  push:
    tags:
      - 'v*'

jobs:
  publish:
    runs-on: ubuntu-latest
    environment: release  # Restricts to release environment secrets
    
    permissions:
      id-token: write  # Allow OIDC token generation
      contents: read
      
    steps:
      - uses: actions/checkout@v4
      
      - name: Setup .NET
        uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'
      
      - name: Build and Pack
        run: dotnet pack src/ElBruno.{ProjectName}/ElBruno.{ProjectName}.csproj -c Release -o artifacts
      
      # CRITICAL: id: login captures OIDC token
      - name: NuGet login (OIDC → temp API key)
        uses: NuGet/login@v1
        id: login
        with:
          user: ${{ secrets.NUGET_USER }}
      
      # CRITICAL: --api-key from login step output
      - name: Push to NuGet.org
        run: dotnet nuget push artifacts/*.nupkg \
          --api-key ${{ steps.login.outputs.NUGET_API_KEY }} \
          --source https://api.nuget.org/v3/index.json \
          --skip-duplicate
```

### NuGet Setup Checklist

- [ ] Created service principal at NuGet.org (username = `{organization}` or personal account name)
- [ ] Created trusted publishing entry at NuGet.org dashboard with exact username
- [ ] Created `release` environment in GitHub repo settings
- [ ] Created `NUGET_USER` secret in `release` environment (value = service principal username, NOT API key)
- [ ] Named workflow file EXACTLY `publish.yml`
- [ ] Used `id: login` on NuGet/login step
- [ ] Used `--api-key ${{ steps.login.outputs.NUGET_API_KEY }}` on nuget push
- [ ] Committed all package assets (icons, images) to git
- [ ] Updated .csproj with correct License metadata (`<PackageLicenseExpression>` instead of `<License>`)
- [ ] Created tag with: `git tag v{VERSION} && git push origin v{VERSION}`

### When Publish Fails (Debugging Reference)

**Error resolution table:**

| Error | Cause | Fix |
|-------|-------|-----|
| `Workflow mismatch for policy 'X'` | Wrong filename (not `publish.yml`) | Rename, DELETE old tag, RECREATE tag |
| `No matching trust policy owned by '__NUGET_USER__'` | Hardcoded placeholder username | Use `${{ secrets.NUGET_USER }}` secret |
| `Could not find a part of the path '/home/runner/.../docs/images'` | Asset not committed to git | Commit assets, DELETE old tag, RECREATE tag |
| `No API key provided in header` | Missing `--api-key` parameter or not captured from login step | Use `id: login` and pass `${{ steps.login.outputs.NUGET_API_KEY }}` |
| `HTTP 401 at token exchange` | Wrong environment or missing permissions | Verify `environment: release` and `permissions.id-token: write` |
| `Metadata does not conform to schema` | Invalid License or package metadata in .csproj | Use `<PackageLicenseExpression>MIT</PackageLicenseExpression>` |

**Recovery process:**
1. Fix the identified issue (rename, update secret, commit assets, adjust API call)
2. Delete the failed tag: `git tag -d v1.0.0 && git push origin --delete v1.0.0`
3. Recreate the tag: `git tag v1.0.0 && git push origin v1.0.0`
4. Monitor workflow run; compare against ElBruno.Text2Image or ElBruno.BM25 v0.5.1 if still failing

## Release to NuGet Workflow (Automated)

### Quick Start: Creating a Release

The release process is **fully automated**. When you push a version tag, the package publishes to NuGet within 2-3 minutes.

**User workflow:**

```bash
# 1. Bump version in .csproj
#    Edit: src/ElBruno.BM25/ElBruno.BM25.csproj
#    Change: <Version>0.5.1</Version>

# 2. Commit version bump + assets
git commit -am "docs: add assets and bump to v0.5.1"

# 3. Create version tag
git tag -a v0.5.1 -m "Release v0.5.1"

# 4. Push commits and tag
git push origin main --tags

# 5. Create GitHub Release (optional but recommended)
gh release create v0.5.1 \
  --title "v0.5.1: Package Improvements" \
  --notes "Release notes from CHANGELOG.md"
```

**That's it.** The workflow handles the rest automatically.

---

### How It Works: The Automated Pipeline

| Step | What Happens | Time |
|------|--------------|------|
| 1. Push tag | You push `v0.5.1` tag to GitHub | 0s |
| 2. Detect | GitHub Actions detects the tag and triggers `publish.yml` | ~5s |
| 3. Extract | Workflow extracts version from `.csproj` (source of truth) | ~10s |
| 4. Tests | Run project tests (optional, skipped if none exist) | ~20s |
| 5. Build | Build Release configuration | ~30s |
| 6. Pack | Create `.nupkg` package file | ~10s |
| 7. Validate | Verify package created successfully | ~5s |
| 8. Auth | Authenticate to NuGet via OIDC token exchange | ~10s |
| 9. Publish | Upload `.nupkg` to NuGet.org | ~20s |
| 10. Index | NuGet indexes package and makes it searchable | ~60-120s |
| **Total** | **~2-3 minutes** | **Fully automated** |

### If Something Goes Wrong: Troubleshooting

**Before you try anything:**
- Always check the workflow logs: **Actions** → **Publish to NuGet** → click the failed run
- Read the red error message in the step that failed
- Use the table below to find the cause

| Symptom | Likely Cause | Fix |
|---------|--------------|-----|
| Build passes, Push fails with `HTTP 401` | OIDC auth failed (missing `id: login` in workflow) | Verify `id: login` exists on NuGet/login step |
| Build fails: `Could not find icon.png` | Asset not committed to git before tagging | Commit assets, delete tag, recreate tag |
| Build fails: `Metadata does not conform to schema` | Invalid License in .csproj | Use `<PackageLicenseExpression>MIT</PackageLicenseExpression>` (not `<License>`) |
| Build fails: `Could not find Version tag in .csproj` | Version tag missing or malformed in .csproj | Verify `<Version>0.5.1</Version>` exists and is not empty |
| Publish fails: `Workflow mismatch for policy` | Workflow file renamed (must be `publish.yml`) | Rename back to `publish.yml`, delete tag, recreate tag |
| Everything mysteriously fails | Tag points to wrong commit | Delete tag, push fixes, recreate tag |

**Recovery steps:**

```bash
# 1. Delete the failed tag (both locally and on GitHub)
git tag -d v0.5.1
git push origin --delete v0.5.1

# 2. Fix the issue (see table above)

# 3. Recreate the tag
git tag -a v0.5.1 -m "Release v0.5.1"
git push origin v0.5.1

# 4. Monitor workflow in Actions tab
```

### Common Questions

**Q: What if I forget to bump the version in .csproj?**

A: The workflow extracts the version from `.csproj`. If the version matches an existing NuGet release, the `--skip-duplicate` flag prevents an error—but the old package is re-published. Always bump `.csproj` before tagging.

**Q: Do I need to manually create a GitHub Release?**

A: No. Pushing the tag is enough to trigger the workflow. Creating a GitHub Release on top of the tag is optional (good for visibility and release notes).

**Q: Can I publish multiple versions at once?**

A: No. Each tag is a separate release. Create multiple tags (`v0.5.0`, `v0.5.1`) and push them separately.

**Q: What happens if the build or tests fail?**

A: The workflow stops and NuGet publishing is skipped. Check the Actions logs for details and fix the issue before recreating the tag.

## Versioning & Releases

### Semantic Versioning

All ElBruno projects follow **Semantic Versioning** (`MAJOR.MINOR.PATCH`):

- **MAJOR** — Breaking changes
- **MINOR** — New features (backward compatible)
- **PATCH** — Bug fixes (backward compatible)

**Example:** `v0.5.0` = Major version 0, Minor version 5, Patch 0

### Release Process

1. **Update version** in `.csproj` and `CHANGELOG.md`
2. **Commit** with message: `chore: bump version to v0.5.0`
3. **Create git tag:** `git tag v0.5.0`
4. **Push tag:** `git push origin v0.5.0`
5. **Monitor workflow:** `.github/workflows/publish.yml` auto-triggers
6. **Verify NuGet:** Package appears on NuGet.org within 1-2 minutes
7. **Create GitHub Release:** Publish release notes (auto-drafted from CHANGELOG)
8. **Announce:** Blog post, LinkedIn, Twitter

### Release Tag Deletion & Recreation

If a release fails and you need to recreate the tag:

```bash
# Delete local and remote tag
git tag -d v0.5.0
git push origin --delete v0.5.0

# Fix the issue (commit assets, update workflow, etc.)
git add .
git commit -m "fix: include assets for release"

# Recreate the tag
git tag v0.5.0
git push origin v0.5.0
```

**Common reasons to recreate:**
- Missing package assets (icons, images)
- Workflow filename mismatch
- OIDC authentication configuration
- Missing/incorrect secret setup

### Pre-Release Versions

For beta/preview versions, use prerelease tags:

- `v0.5.0-beta.1` — First beta
- `v0.5.0-rc.1` — Release candidate
- `v0.5.0-preview.1` — Preview

Mark as pre-release on NuGet.org and GitHub.

## Publishing Lessons Learned (v0.5.0 Release — ElBruno.BM25)

This section captures critical lessons from the ElBruno.BM25 v0.5.0 NuGet publishing process. **Read this before attempting your first publish.**

### Critical Lesson: Use `NuGet/login@v1` Action

**❌ WRONG — Will fail with 403 Forbidden:**
```yaml
- name: Obtain OIDC token
  uses: actions/github-script@v7
  with:
    script: |
      const token = await core.getIDToken('https://api.nuget.org');
      core.setOutput('token', token);
- name: Publish to NuGet
  run: dotnet nuget push nupkg/*.nupkg --api-key "${{ steps.oidc.outputs.token }}" --source https://api.nuget.org/v3/index.json
```

**✅ CORRECT — Use NuGet/login action:**
```yaml
- name: NuGet login (OIDC → temp API key)
  uses: NuGet/login@v1
  id: login
  with:
    user: ${{ secrets.NUGET_USER }}
- name: Publish to NuGet
  run: dotnet nuget push nupkg/*.nupkg --api-key ${{ steps.login.outputs.NUGET_API_KEY }} --source https://api.nuget.org/v3/index.json --skip-duplicate
```

**Why:** The `NuGet/login@v1` action:
- Obtains OIDC token with correct audience (handled internally)
- Converts token to temporary NuGet API key
- NuGet recognizes the temp key and validates against trusted publishing policy
- Eliminates 403 errors from manual token handling

### NuGet Trusted Publishing Setup (Prerequisites)

**On NuGet.org (one-time setup):**
1. Go to: https://www.nuget.org/account/trustedpublishers
2. Create trusted publishing policy:
   - **Service Principal:** Your NuGet username (must match `NUGET_USER` secret)
   - **GitHub Repository:** `github.com/{owner}/{repo-name}`
   - **Workflow:** `publish.yml` (MUST be exact filename)
   - **Environment:** `release`
3. Verify policy status: Must show "Active"

**On GitHub (one-time setup):**
1. Settings → Environments → Create "release" environment
2. Add secret in release environment:
   - **Name:** `NUGET_USER`
   - **Value:** Your NuGet service principal username

**In `.github/workflows/publish.yml` (exact structure):**
```yaml
on:
  push:
    tags:
      - 'v*'

jobs:
  publish:
    runs-on: ubuntu-latest
    environment: release              # ← Required
    permissions:
      id-token: write                # ← Required for OIDC
      contents: read
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-dotnet@v4
        with:
          dotnet-version: '8.0.x'
      - run: dotnet restore {ProjectPath}
      - run: dotnet build {ProjectPath} --configuration Release --no-restore
      - run: dotnet pack {ProjectPath} --configuration Release --output nupkg
      - uses: NuGet/login@v1
        id: login
        with:
          user: ${{ secrets.NUGET_USER }}
      - run: dotnet nuget push nupkg/*.nupkg --api-key ${{ steps.login.outputs.NUGET_API_KEY }} --source https://api.nuget.org/v3/index.json --skip-duplicate
```

### GitHub Actions Best Practices

**1. Explicit Project Paths**
- ❌ `dotnet restore` (fails in CI, ambiguous)
- ✅ `dotnet restore src/ElBruno.BM25/ElBruno.BM25.csproj` (explicit)

**2. Action Versions**
- ✅ Use specific major versions: `@v4`, `@v1`, `@v3`
- ❌ Never use `@latest` or floating versions

**3. Job Permissions**
```yaml
permissions:
  id-token: write  # For OIDC token generation
  contents: read   # For repo access
```
- Always declare minimum permissions needed
- Never use `permissions: write-all` (security risk)

**4. Environment-Scoped Secrets**
- Store publish-only secrets in `release` environment
- They are NOT exposed to other workflows/jobs
- Ref: `${{ secrets.NUGET_USER }}` (auto-scoped to job's environment)

### Troubleshooting Publish Failures

| Error | Cause | Fix |
|-------|-------|-----|
| `403 Forbidden` | Manual OIDC token handling | Use `NuGet/login@v1` action |
| `Policy not found` | No trusted publishing policy on nuget.org | Create policy at https://nuget.org/account/trustedpublishers |
| `Unauthorized` | `NUGET_USER` secret not set in release environment | Create secret in GitHub release environment |
| `Workflow filename mismatch` | Workflow named `publish-nuget.yml` instead of `publish.yml` | Rename to exactly `publish.yml` (NuGet enforces this) |
| `Policy inactive` | Policy created but not activated | Go to nuget.org, click "Activate for 7 days" |
| `Missing OIDC token` | Job doesn't have `permissions: id-token: write` | Add to job permissions |

### CI Performance Tuning

**Issue:** Performance tests timeout in CI (GitHub Actions runners ~1000x slower than local dev).

**Solution:** Environment-specific timeouts:
```csharp
// Relax timeouts for CI environment
const long CI_SAVE_TIMEOUT_MS = 600_000;   // 10 minutes for CI
const long CI_LOAD_TIMEOUT_MS = 300_000;   // 5 minutes for CI
// Local performance still <1s; CI needs buffer for I/O-heavy operations
```

**Key takeaway:** Don't assume CI speed matches local. Build in 10x+ buffer for serialization, large allocations, I/O operations on cloud runners.

### Documentation Reorganization

**Best practice:** Reorganize docs EARLY in project lifecycle, before public release.

**Process:**
1. Create `docs/guides/`, `docs/architecture/`, `docs/api/` subdirectories
2. Move or copy README.md, LICENSE into `docs/`
3. Update ALL markdown links to reflect new paths
4. Search for old patterns: `/docs/old-path` → `/docs/new-path`
5. Test links in rendered markdown (GitHub, NuGet)
6. Include "docs: reorganize folder structure" in commit message

**Lesson:** Link migration is error-prone and hard to debug later. Do it early, commit with clear messages.

### Disabled GitHub Workflows

**Keep only essential workflows:**
- `build.yml` — Build, test, run benchmarks (keep)
- `publish.yml` — Publish to NuGet on tag (keep)
- `squad-*.yml` — Team automation (disable if not using Squad)
- Others — Remove or disable

**Reason:** Unnecessary workflows add noise to CI/CD pipeline and slow down your build matrix.
